title: "Generating g-code with Inkscape"

vplotter reads standard g-code from stdin. The recommended workflow for producing g-code from artwork is to draw or import your design in [Inkscape](https://inkscape.org) and then export it using the [gcodetools extension](https://github.com/cnc-club/gcodetools). This gives you direct control over feed rates, Z values, and output file location.

## Required tools

- [Inkscape](https://inkscape.org) — open-source vector graphics editor available for Linux, macOS, and Windows
- [gcodetools extension](https://github.com/cnc-club/gcodetools) — Inkscape extension that converts vector paths to g-code; follow the installation instructions in its repository

## Workflow

<Steps>
  <Step title="Create or import your artwork">
    Open Inkscape and draw your design, import an SVG, or use **Path > Trace Bitmap** to convert a raster image into paths.
  </Step>

  <Step title="Convert to paths">
    Select all objects and choose **Path > Object to Path**. gcodetools only processes path elements; shapes and text must be converted first.
  </Step>

  <Step title="Set document units to millimeters">
    Open **File > Document Properties** and set the display units to **mm**. Your document dimensions should match the physical size of your drawing canvas so coordinates map correctly.
  </Step>

  <Step title="Configure gcodetools">
    Open **Extensions > Generate from Path > Path to Gcode**. Set your feed rate, Z-up and Z-down values (these correspond to `--z_up` and `--z_down`), and choose an output directory for the `.ngc` file.
  </Step>

  <Step title="Generate the g-code file">
    Run the extension. gcodetools writes an `.ngc` file to the directory you specified, containing one or more toolpath sequences.
  </Step>

  <Step title="Run vplotter with the generated file">
    Pipe the `.ngc` file into vplotter using stdin redirection:

    ```bash
    sudo vplotter --x0=170 --y0=-720 --baselength=685 --z_up=11 --z_down=7 --steps=40.1 < output.ngc
    ```
  </Step>
</Steps>

<Tip>
  Set your Inkscape document dimensions to match the physical size of your canvas in millimeters before generating g-code. If the document is sized differently, your artwork will be scaled or clipped when plotted.
</Tip>

## G-code file format

gcodetools produces standard RS-274 g-code. vplotter processes the file line by line and acts on the commands it recognizes. Here is a breakdown of the structure using the example from the vplotter test file (`test.ngc`):

| Command | Meaning |
|---|---|
| `%` | Program start/end marker |
| `(comments)` | Parentheses-enclosed comments; ignored by vplotter |
| `M3` | Start spindle; emitted by gcodetools but ignored by vplotter |
| `G21` | Set units to millimeters; always included by gcodetools |
| `G00 Z5` | Rapid move to lift the pen before repositioning |
| `G00 X... Y...` | Rapid move to the start of a new path segment |
| `G01 Z-0.125 F100.0` | Feed move to lower the pen onto the canvas |
| `G02` / `G03` | Clockwise / counter-clockwise arc move along the path |
| `M2` | End of program (emitted by gcodetools; ignored by vplotter) |
| `M30` | End of program (handled by vplotter: lifts pen, returns to origin, exits) |

The following is the opening sequence from a real gcodetools-generated file:

```gcode
%
(Header)
(Generated by gcodetools from Inkscape.)
G21 (All units in mm)

G00 Z5.000000
G00 X48.703383 Y71.271882

G01 Z-0.125000 F100.0(Penetrate)
G02 X47.184429 Y63.714616 Z-0.125000 I-19.559355 J0.000000 F400.000000
G02 X44.901282 Y62.195661 Z-0.125000 I-2.283150 J0.956429
G00 Z5.000000
```

<Warning>
  The `M3` (start spindle), `M5` (stop spindle), and `M2` (end program) commands that gcodetools includes in its output are silently ignored by vplotter. Only `G0`, `G1`, `G2`, `G3`, `G20`, `G21`, and `M30` are handled. To end a plot cleanly, your g-code must include `M30`.
</Warning>
